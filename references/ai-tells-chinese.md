# 中文 AI 腔 Catalog（去 AI 味 · 说人话）

把中文文本从"像模型在表演写作"拉回"像具体研究者在当前场景下表达"。提炼自 `shuorenhua`。
核心立场：**先保信息，再谈风格**。任何改写都不能新增事实、删核心事实或改变责任主体。这不是敏感词替换器，
也不反技术、反抽象、反专业——目标是减少模板感、表演感和语域漂移。

## 执行顺序（固定，不跳步）

1. **判场景**：`chat / status / docs / public-writing`（学术稿多为 docs 或 public-writing 的严谨变体）
2. **划 protected spans**：先锁必须保留的术语、系统主语、引用原文、命令、报错、数字、公式、引用键
3. **判 Tier**：Tier 1 / 2 / 3（问题命中强度，不等于改写力度）
4. **判档位**：`minimal / standard / aggressive`
5. **判 scope**：`structural / bounded / in-place`（能删到什么程度）
6. 先按"模式"处理，再按"词条"兜底
7. **两步回读**：先保真回读，再残留味回读
8. **输出**：默认单一推荐版本；用户要"先标问题"时切 annotation mode

## 场景与默认档位

| 场景 | 信号 | 默认档位 |
|------|------|----------|
| `chat` | 短回复、日常沟通、评论、即时反馈 | minimal |
| `status` | 站会更新、进度同步、复盘摘要、汇报 | minimal / standard |
| `docs` | 操作/技术说明、FAQ、事故复盘、**论文正文/方法** | minimal（重可检索、可复现、术语稳定） |
| `public-writing` | 公众号、对外文章、观点写作、**摘要/引言的对外表达** | standard |

## Edit scope（能不能删句/段，与档位是两条轴）

- `structural`（默认，短文本/明确要重写）：可删整句空总结、并句、轻调句序。
- `bounded`（中文长文 ≈1000字+ 默认）：句内去味照做；但**整句删除只进"建议删除（待确认）"清单**，
  每条写「原文 + 为什么删了不丢信息」，长度由作者拍板。避免 structural 不可控缩水（同篇可能 −18% 或 −39%）。
- `in-place`（用户要求严格保句数/一句不删）：整句空话也不删，只做句内降调；删短语前先做语义独立性检查，
  删后剩余必须仍是完整、无悬空指代的陈述句。

## 高频 AI 腔（按类处理，不必逐词命中）

- **开场套话 / 元评论**：`值得注意的是`、`需要指出的是`、`众所周知`、`让我来为你解释`、`希望这对你有帮助` → 删。
- **空总结 / 收束腔**：`综上所述`、`总而言之`、`归根结底`、`本质上`、`最终来看` → 删或压回事实句。
- **二元对比骨架**：`不是 X，而是 Y`、`与其 X，不如 Y` → 多数删前半，直接说 `Y`。
- **无源权威铺垫**：`研究表明`、`数据显示`、`业内人士认为`、`大量实践证明` → 见下方三模式；**不补虚构来源**。
- **商业黑话 / 表演性工程师腔**：`赋能`、`抓手`、`闭环`、`收窄`、`兜住`、`落盘`、`打法`、`颗粒度`、`心智` → 改回普通动作。
- **过度承接 / 替读者做判断 / 身份认证式夸奖**：`你问到了问题的核心`、`这正是顶刊作者的素养`、`你只是……` → 删姿态层，回到具体判断。
- **翻译腔**：长定语链、被动堆砌、`基于……`、`通过……来……`、`……的一个……` → 缩短主语和动作。
- **渲染性修饰扎堆（Tier 2）**：`深度`、`维度`、`生态`、`范式`、`纵深`、`系统性`、`全方位` 同段 2–3 个才处理，保留最贴切的一个。
- **泛化拔高（Tier 3）**：`重要 / 关键 / 核心 / 提升 / 显著 / 创新` 只在全文密度过高时替换一部分。
- **破折号 / 硬造金句**：中文破折号滥用同英文——按需换成句号、逗号、冒号或括号；不要每句都想落一句"金句"。

## 无源引用：固定三模式选一（学术稿尤其关键）

- `rewrite-safe`：直接删掉 `研究表明 / studies show / 业内人士认为` 类铺垫，只保留本就成立、不依赖虚构来源的判断。默认用于 `chat` / `public-writing`。
- `audit-only`：不替作者补来源，也不把无证据判断写成像有证据；明确指出"这里缺来源/缺归属"，必要时保留原句。默认用于 `docs` / `status` / **论文正文**。
- `rewrite-with-placeholder`：仅当用户明确要保留原结构/语气时用；可写"有研究认为……，但此处未给出处"，**不能补具体机构、数据、年份、研究名称**。

跨场景时取更保守的 `audit-only`。学术写作里，无源引用是硬伤——优先 audit，让作者补真实文献，而不是让模型编。

## Protected spans（默认保留，除非用户明确要改风格且不损信息）

引用原文、命令、接口名、参数名、字段名、配置项、日志、报错；技术文档里的系统行为主语；
postmortem / incident / PRD / release note 中的专业术语；**数字、统计量、p 值、公式、单位、引用键（cite key）**；
承载关键事实的抽象句（即使"有点像 AI"）。中英混排句里的英文词按当前句实际语义判断，不机械套英文词表。

## 两步回读

**Pass 1 | 保真回读（必做）**：① protected spans 有没有漂 ② 信息是否丢失 ③ 语域是否统一 ④ 术语是否失真
⑤ 删改后有没有生硬断裂。删掉一句后段落没了落点，用原文已有信息重组一条事实句，**不补口号句**；找不到就让段落短一点。
`bounded/in-place` 额外查：每个信息点（事实/数字/判断/动作）在输出里可追溯；`in-place` 输出低于原文 85% 字数要回退查是否误删整句。

**Pass 2 | 残留味回读（保真后才做）**：查 ① 开场残留 ② 总结残留 ③ narrator 残留（还在解释"这说明了什么"）
④ 空泛判断残留（`方向是对的 / 意义重大`）⑤ 句长过匀。只做轻量修正，不重写全文、不补事实、不动术语/参数/命令/责任归属。
`docs / status` 默认更保守——第二遍会让语气变口语或影响保真就停在第一遍。

## 正向目标（改成什么样才算"更像人"）

有具体信息（不靠空洞总括撑气势）· 有主语和动作（不靠虚假主体兜底）· 语域统一（不在技术腔/商业腔/自媒体腔之间跳）·
以"可直接发/可投稿"为终点（不为更像人抛光到失真）· 节奏来自删冗余而非硬造金句 · 立场来自判断或事实而非"故作洞见"·
有边界，没把握就直说。

## 误杀防护

引用原文、命令、接口名、字段名、日志、报错、系统主语、技术报告术语默认保留。专业文本可以专业——关键是别模板化、别表演化，
不要为了"像人"把严谨的学术表达改得更假。判断靠**扎堆**，不靠单个词。
